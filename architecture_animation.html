Memory & audit
A message through the stack
1
You type a question
Streamlit captures your input and immediately writes it to PostgreSQL — before anything else runs.
save_msg("user", prompt)
2
Memory is loaded
The last 15 messages for your session are fetched from PostgreSQL. The agent sees full context — not just the current question.
load_history(session_id)
3
LangGraph reasons
Your message + history + SYSTEM_PROMPT is sent to Mistral via Ollama. The model reads the rules and decides which tool to call.
agent.invoke({"messages": history})
4
Tool executes + logs
The chosen tool runs — ChromaDB search, Excel read, API call, web search. Every call is logged to the audit table before returning.
_log_tool(name, input, output)
5
Response rendered and saved
The model reads the tool result and writes a natural language response. Streamlit renders it. PostgreSQL stores it permanently.
save_msg("assistant", response)
What this actually taught me

Models are swappable components. LLaMA → Mistral → Gemma → back. One line in agent.py. Tools, memory, RAG — unchanged.
RAG quality determines everything. The difference between an AI that hallucinates and one that reasons over your actual data is the embedding model, chunking strategy, and domain separation.
Architecture decisions compound. Standard SQL meant SQLite → PostgreSQL in one afternoon. Modular tools meant adding a new capability without touching the agent. Good decisions in week 1 pay off in week 3.
Audit by design, not afterthought. Every tool call logged before execution. Every web search recorded. Every file operation tracked. Trust in AI systems comes from the infrastructure around the model — not from the model itself.
Small local models have real limits. 7B parameters runs locally with zero cost and zero data leaving your machine. But tool calling reliability requires either larger models or cloud APIs. Understanding why that ceiling exists is more valuable than pretending it doesn't.
